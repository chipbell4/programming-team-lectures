More Introducing

This "class" is essentially competitive programming. You're given a problem set, asked to solve each problem by writing a program to generate a given output from a given input. We the judges run your code with separate input and see if your algorithm is correct and efficient.

It's as easy as that ™.

Strategies

There's lot's of different things to think about here. But I've listed just a few that I could think of. (Press the down arrow).

Do the easy problems first
Leverage Your teammates
Code on paper

Don't hog the computer thinking, while someone could be coding

Know your language

Don't waste time searching the docs frantically for "how to sort a list", etc. You should spend your time either thinking about the algorithm or coding it.

Don't code your initial reaction

Sometimes problems are more complex (or simple) than you think. Think about the large cases and if your algorithm can handle it. Sometimes talking with a teammate can find these fatal flaws (see here)

Know your algorithms

Many problems are textbook implementations of famous algorithms. Others are tweaks on famous algorithms. Learn how to recognize this and run with it.

Know how to bring your algorithms

Never code a common algorithm in a competition. You're prone to bugs and good old fashioned coder frustration. Instead bring well-tested implementations of common data structures and algorithms (e.g. Graphs, BFS, Dijkstra's, etc.) on paper and copy them by hand into the computer.

Examples

Here's an example to illustrate the point...

Triangle Numbers

Triangle numbers are defined as $T(n) = \sum_{k=1}^{N} k$. If I give an $n$, calculate $T(n)$. 

int N = 10;
int total = 0;
for(int i = 1; i <= N; i++) {
	total += i;
}
System.out.println(total);
Easy right?
  • What if $n = 10^2$?
  • What if $n = 10^4$?
  • What if $n = 10^8$?
  • What if... You get the idea.

What sort of things should we be considering?

  1. Truncation: When will we run out of space in a 32-bit integer
  2. At what point will our algorithm be too slow (our current one is $O(n)$?
How can we do better?
  • Represent integers with more space: unsigned long or perhaps BigInteger. Python folks, carry on.
  • Find a closed form for this thing and get rid of the summation. Anyone know the formula (and the story about Gauss?)
This is the formula

$T(n) = \frac{n (n+1) }{2}$
